# Hand-off: hand-author test specs for scenario1–4 (with close + autofill behavior)

You are the next agent. Your job has two halves, and you own both:

1. **Normalize a specific lifecycle across all four production scenarios**
   (`scenario1`, `scenario2`, `scenario3`, `scenario4`) so every one of them
   behaves the same way with respect to seating:
   - As soon as the scenario starts accepting players, it closes to new
     joins (no more hub-transfers in). Use `ctx.closeScenario()`.
   - Exactly **10 seconds** (sim-time) after closing, any remaining seat
     shortfall is filled with bots by calling `ctx.spawnBot(...)`. Fill up
     to the scenario's **minimum required player count** — see
     "min_player_count / maxPlayers" below for where that number comes
     from per scenario.
   - Once closed, the scenario **never re-opens**. No code path may call
     `closeScenario()` again for the sake of re-opening, and no code path
     may relax the closed state. The only exception is the transient
     open/close toggle the hub-transfer **swap-connection** flow does while
     a swap is in flight — that's framework behavior and is fine; you must
     not interfere with it.

2. **Hand-author test specs** for each of the four scenarios that
   demonstrate the behavior above. These are not generated by an LLM; write
   the `spec.json` files yourself. One spec per scenario is the minimum —
   additional specs per scenario are welcome if they cover meaningful
   outcome branches (e.g. scenario1's majority vote, scenario2's all-movers
   vs all-stayers) that already exist.

Do NOT delegate the whole thing to `create-scenario-agent` or
`direct-agent`. Edit scenarios in place; write specs in place. Use the
sub-agents only when you hit something you legitimately can't hand-solve.

## When to close

"Close" here means: stop accepting new hub transfers into this scenario.
The canonical hook is to call `ctx.closeScenario()` from
`onPlayerConnect` as soon as the scenario knows it has at least one
player — i.e. on the very first connect. The ten-second fill timer
starts from that same event (`ctx.after(10_000, 'fillBots')`). This has
two properties you want:

1. A single-player join closes the room immediately. 10 s later, 3 bots
   fill it. No race between "another human arriving" and "bot fill
   firing."
2. The swap-connection toggle inside the framework is orthogonal; it
   flips the accept flag momentarily to receive a swap target, then
   re-closes. You do not call any API to influence this — just don't
   unconditionally re-open the room from your script.

The exact pattern to copy — straight from `scenario2`, but simplified
because you're not gating on room-entry:

```ts
const BOT_FILL_DELAY_MS = 10_000

interface State { closed: boolean; fillScheduled: boolean }

const script: GameScript<State> = {
  initialState: () => ({ closed: false, fillScheduled: false }),

  onPlayerConnect(state, ctx) {
    if (state.closed) return
    state.closed = true
    ctx.closeScenario()
    if (!state.fillScheduled) {
      state.fillScheduled = true
      ctx.after(BOT_FILL_DELAY_MS, 'fillBots')
    }
  },

  handlers: {
    fillBots(_state, ctx) {
      const needed = MIN_PLAYERS - ctx.getPlayerIds().length
      for (let i = 0; i < needed; i++) ctx.spawnBot(FILL_BOT)
    },
    // ... whatever else the scenario's real gameplay handlers are
  },
}
```

Integrate this block into each of `scenario1/scenario.ts`,
`scenario2/scenario.ts`, `scenario3/scenario.ts`, `scenario4/scenario.ts`
without breaking their existing gameplay handlers. For `scenario2` this
means *replacing* the current "fill on first-room1-entry" trigger with
the simpler "fill 10 s after first connect" trigger — update
`scenario2`'s existing test specs (`all-movers`, `all-stayers`) if the
timing shift breaks them.
