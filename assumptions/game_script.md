# Game Script — Assumptions

- `GameSpec` and `GameScript` types are defined in `src/game/GameSpec.ts` (client) and `server/src/GameSpec.ts` + `server/src/GameScript.ts` (server). The GameSpec definition is kept identical between both copies; a diff is a bug.
- Vote region assignment is computed by Euclidean distance: a player is inside region `r` when `hypot(px − r.x, pz − r.z) ≤ r.radius`.
- Only one region is assigned per player at a time; if a player is simultaneously within multiple active regions, the first one found in iteration order wins.
- The `onVoteChanged` callback fires only when a player's region assignment changes, not on every move. Callbacks are per-listener, not per-region: one `onVoteChanged` call registers one listener covering all supplied region ids.
- The assignments map passed to `onVoteChanged` callbacks contains every currently tracked player (all connected human players), not just those in the specified regions.
- `after` is tick-based: the delay is converted to a target server tick via `ceil(ms / 50)` (minimum 1 tick) and fires when `Room.runTick` reaches that tick. The returned cancel function marks the queued entry cancelled so it is dropped on the next drain. It does not use `setTimeout` — nothing in the game-script path does.
- `onPlayerReady` is optional on `GameScript`. If the script omits it, the ready signal is recorded on both the room (`readyPlayerIds`) and the scenario (for replay by `start()` on a created-not-started scenario) but never reaches the script.
- An `instruction` message received by the client is converted into a `ShowRuleEvent` whose rules come from the message's `lines` field (each line carries `{label, text}`). The label is whatever the authoring instruction spec declared (commonly `RULE`, `COMMAND`, or `FACT`) — it is not hardcoded to `COMMAND`. The rule popup is shown immediately (no 250 ms buffer delay) because the `instruction` message is sent directly to the target player and not through the broadcast path.
- Calling `eliminatePlayer` from within an `onVoteChanged` or `after` callback is safe: `Room.removePlayer` is re-entrant (map deletions on already-absent keys are no-ops).
- Each `Scenario` added to a `MultiplayerRoom` is constructed with a fresh script instance via `ScenarioSpec.scriptFactory()` — `DefaultScenarioOrchestration` calls it once per `createRoom`. This guarantees no state from a prior scenario (e.g. `connectCount`, listener flags, timer handles) leaks into a new scenario for the same `ScenarioSpec`.
- Floor geometry visibility is tracked per-player in `Scenario`. On `onPlayerAttach`, each player receives the initial state derived from `initialVisibility` in the scenario spec; any id absent from `initialVisibility` defaults to visible (`true`). Vote regions absent from `initialVisibility` default to inactive.
- Walkable area variants are evaluated only when geometry is updated globally (all players); per-player visibility changes do not trigger variant re-evaluation.
- Walkable snap uses axis-aligned nearest-point clamping: for each walkable rect, the closest point is `(clamp(px, cx−hw, cx+hw), clamp(pz, cz−hd, cz+hd))`; the globally nearest point across all rects is chosen. This guarantees the player moves to the interior of their cage (not through the wall to the exterior) only when cage geometry is authored so the player's XZ position at vote time is already enclosed by the new walkable rects.
- `onPlayerConnect` callbacks are fired from `Scenario.onPlayerAttach` (immediately if the scenario is started; otherwise deferred and replayed in attach order by `start()`); `onPlayerDetach` is not exposed to the script, but the player is removed from all tracked maps and future `getPlayerIds()` calls will not include them.
- `ScenarioSpec.requiredRoomIds` uses the scoped form `{mapInstanceId}_{localRoomId}`; any id in that list not present in the scenario's attached room set throws from the `Scenario` constructor. `DefaultScenarioOrchestration` passes `room.addMap(map)`'s return value as the attached set, so missing ids surface at room construction before any player can connect. An absent or empty `requiredRoomIds` skips the check entirely.
- `ScenarioSpec.initialRoomVisibility` keys are scoped room ids; `setRoomVisible` takes scoped room ids; the `onPlayerEnterRoom` callback fires with `(playerId, scopedRoomId)`.
- `Scenario.onPlayerMoved` calls `world.setPlayerRoom(playerId, newRoomId)` every time the composed `getRoomAtPosition` (exposed to the scenario via the room, and in turn composed across every `addMap`-registered map) reports a new scoped room id for that player. This keeps `World.playerRoom` (consumed by `World.getAccessibleRooms`) aligned with what the script observes. A `getRoomAtPosition` return of `null` is treated as "no transition" — the stored room is not cleared.
- The current deployment has exactly one world and one map instance registered via `world.addMap(gameMap)`, with exactly one attached scenario marked as the default-open scenario. The scenario stays open (accepting auto-attach on new connections) until `closeScenario` is called; the `ScenarioManager` supports multiple concurrent scenarios per room, but `DefaultScenarioOrchestration` ships with exactly one.
- The spec permits per-player accessible-rooms subsets, but no code path currently calls `World.setAccessibleRoomsOverride`; movement containment relies on the walkable area and floor-geometry toggles, not on the accessible-rooms set.
