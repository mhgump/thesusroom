# Serverâ€“Client Protocol â€” Assumptions

- The client-side `src/game/World.ts` is the single source of truth. The server's `server/src/World.ts` is a one-line re-export (`export * from '../../src/game/World.js'`) that resolves to the same module at runtime; a divergence between the two is impossible by construction rather than enforced by a sync step.
- Each websocket room maps 1:1 to one `MultiplayerRoom` instance and one `World` instance, hosting one map instance and one attached scenario marked as the default-open scenario. The `ScenarioManager` supports multiple concurrent scenarios per room, but `DefaultScenarioOrchestration` ships with exactly one.
- `mapInstanceId` equals the scenario id for every shipped scenario (e.g. `demo`). Room ids on the wire are always the scoped form `{mapInstanceId}_{localRoomId}` (e.g. `demo_room1`). This applies to `room_visibility_state.updates[].roomId`, the keys of `ScenarioSpec.initialRoomVisibility`, `ScenarioSpec.requiredRoomIds`, and the `roomId` argument of `onPlayerEnterRoom` callbacks.
- `MultiplayerRoom.addMap(gameMap)` is called from the orchestration immediately after the `MultiplayerRoom` constructor returns, before any scenario is built. It calls `world.addMap(gameMap)` (which internally builds scoped room ids + default adjacency via `buildMapInstanceArtifacts` and registers the resulting `WorldMapInstance` on the world), spawns the map's NPCs, composes the map's `getRoomAtPosition` into the room-level lookup exposed to scenarios, and returns the scoped room ids contributed by the map for use when constructing scenarios.
- The only routing-key shape the server accepts on the non-observer path is `r_{scenario_id}`. The empty path and `/` are aliased to `r_demo`; anything else is rejected with close code 4004. Scenario ids are lowercase identifiers; `{scenario_id}` may not be empty.
- When multiple open rooms exist for the same routing key, the router selects one uniformly at random per connection (`Math.random()` against `openRooms[key].length`). No affinity, skill, or fill-order logic â€” the orchestration is responsible for keeping the open list to the size that actually makes sense for its mode (the default mode caps the list at one).
- The per-key open-room list is maintained as rooms open and close: rooms are appended on creation and removed on orchestration close-callback. The router also lazily prunes any room whose `orchestration.isOpen` returns false when it goes to pick one, as a defence against a callback not firing.
- Observer URL indices are stable per routing key: `{i}` indexes into the per-key all-rooms list; slots freed by room destruction are reused by later rooms (the router picks the lowest-index null slot before appending).
- `ready` is a `ClientMessage` variant with no payload (`{ type: 'ready' }`). Player connections route it to `MultiplayerRoom.handlePlayerReady(playerId)`, which adds the player to the room's insertion-ordered `readyPlayerIds` set and forwards to `ScenarioManager.forPlayer(playerId)?.onPlayerReady(playerId)`. The scenario maintains its own insertion-ordered ready set for replay purposes. Observer connections fire `GameServer`'s `observerReadyListeners` set â€” consumers subscribe via `onObserverReady(cb)` which returns an unsubscribe function.
- Scenario lifecycle is created â†’ started â†’ deleted. Pause is not a state. A scenario added to the `ScenarioManager` without `start()` is in the created-not-started state: `onPlayerAttach` still initialises per-player geometry / room-visibility / button state and sends the corresponding messages, but the script's `onPlayerConnect` / `onPlayerReady` callbacks are suppressed and attached + ready ids accumulate in insertion-ordered sets. `MultiplayerRoom.startScenario(id)` calls `Scenario.start()`, which replays `script.onPlayerConnect` once per attached player (in attach order) then `script.onPlayerReady` once per ready player (in ready order). `start()` is one-way and idempotent â€” a second call returns immediately. `MultiplayerRoom.deleteScenario(id)` (and the automatic `ScenarioManager.destroyAll` invoked on room teardown) detaches every attached player then calls `Scenario.delete()`; scheduled callbacks queued before deletion but dispatched afterward are dropped via the scenario's alive-gate on the per-scenario schedule wrapper.
- `DefaultScenarioOrchestration` starts its default-open scenario immediately on creation unless the `autoStartScenario: false` option is passed â€” the run-scenario harness passes it so the scenario stays created-not-started while bots connect + auto-ready + the observer browser is prewarmed, then calls `room.startScenario(scenarioId)` once the observer is recording.
- `ctx.closeScenario()` clears the scenario from the `ScenarioManager`'s default-open slot (no new connections auto-attach to it) and, if no default-open scenario remains on the room, fires the orchestration's `onClose` callback so the router drops the room from its open list. The scenario itself keeps running for its already-attached players until the room is destroyed (last player leaves a closed room), at which point `ScenarioManager.destroyAll()` deletes every scenario before the room's `onRoomDone` callback fires.
- The canonical sim-time-per-tick is exactly 50 ms (`TICK_RATE_HZ = 20` in `World.ts`). This is the fixed `dt` passed into `world.processTick` and the unit for every tick-scheduled callback; it never changes.
- The wall-clock tick rate is configurable per `MultiplayerRoom` via the `tickRateHz` constructor option (default `TICK_RATE_HZ = 20`). Production runs at 20 Hz (wall = sim). The test harness (`server/scripts/run-scenario.ts`) defaults to 240 Hz so scenarios execute 12Ă— faster in wall-clock time while preserving identical sim-time behaviour.
- The tick loop uses a drift-corrected self-rescheduling `setTimeout` chain keyed on `performance.now()`, not `setInterval`, so per-tick jitter doesn't accumulate at high tick rates. The current scheduled tick time is stored as an absolute `performance.now()`-stamped `nextTickAt` and advanced by the wall-clock interval after each tick.
- The client's `tickRateHz` arrives in the `welcome` message. `positionBuffer.setServerTickRateHz` records it and `advanceRenderTick` uses it in place of `TICK_RATE_HZ` so remote-player interpolation keeps pace when the server is running faster than 20 Hz. `getServerTickRateHz` is also consulted by `useWebSocket.ts` when deferring `game_event` delivery to the render tick.
- `MultiplayerRoom.scheduleSimMs(ms, cb)` queues a callback to fire on the next tick whose cumulative sim-time has advanced by `ms`; the conversion is `targetTick = serverTick + max(1, ceil(ms / 50))`. The scheduler is drained at the end of `runTick`, after all per-player event emission. Each `Scenario` wraps this in `scheduleScoped` which adds an alive-gate, so callbacks queued before `delete()` but due to fire afterward are dropped at dispatch time.
- Any wall-clock-driven timer that used to call `setTimeout` inside the server core now goes through `scheduleSimMs`: `Scenario.ctx.after`, `ButtonManager.startCooldown`, and `NpcManager` periodic triggers. None of the three consults `Date.now` / `setTimeout` directly for timing.
- The client's remote-render buffer is exactly 5 ticks (250 ms at 20 Hz; scales in wall-clock terms with the server's tick rate); `render_tick` trails `server_world_tick` by this buffer under normal playback.
- The remote-playback catch-up ramp uses exactly these knees: speed is 1.0 while lag < `1.5 Ă— 5` ticks, ramps linearly to 2.0 as lag grows to `3 Ă— 5` ticks, and is clamped at 2.0 above that. `render_tick` is never advanced past `server_world_tick âˆ’ 5`.
- Remote position snapshots are trimmed to a 40-tick window (~2 s); snapshots with `tick < server_world_tick âˆ’ 40` are discarded.
- The correction threshold is exactly 1.6 mm (`CORRECTION_THRESHOLD = 0.0016` in `Player.tsx`). Reconciliation is skipped when the ack position and the recorded predicted end-of-tick position agree within this threshold; when reconciliation runs, the rendered visual is snapped only if the post-replay world position and the currently rendered position differ by more than the same threshold.
- An ack whose `serverTick` is not strictly greater than the most recently reconciled `serverTick` does not drive reconciliation, but its `damage` events still fire and its `clientTick` is still pruned from input history.
- The server never drops or rejects client moves; moves received between ticks are buffered in arrival order, sorted by `clientTick` on the next tick, flattened into a single input array per player, and handed to `world.queueMove` before `world.processTick` runs.
- When multiple moves from the same player are processed in one server tick, every ack sent back carries the same end-of-tick `(x, z)`; events attach only to the final ack so the client applies each event exactly once.
- The client's input history (`tickInputs`, `predictedPosPerTick`) is unbounded: entries are pruned only when their `clientTick` is acknowledged. There is no fixed cap â€” if the server stops acking for an extended period the maps will grow until connection is reset.
- Client ticks reset on reconnect: a reconnecting client receives a new `playerId`, triggering fresh `clientPredictiveTick` and `clientWorldTick` initialisation on both sides.
- Position snapshots and event queues are keyed by integer `serverTick`, not by wall-clock time. `positionBuffer.ts` consults neither `Date.now()` nor `performance.now()` for any gating decision; its only timing input is the per-frame `delta` passed into `advanceRenderTick`.
- `positionBuffer.ts` is a plain module with no React dependency; remote position and event buffer state must not be stored in Zustand (doing so would trigger a React re-render on every frame per remote player).
- `advanceRenderTick` must run at `useFrame` priority âˆ’3 so it updates `renderTickFloat` before any consumer (Player, RemotePlayers at default priority) reads it this frame.
