# Scene — Assumptions

- Coordinate system: X = east (+right), Y = up, Z = south (+toward camera); floor at Y = 0.
- `CAPSULE_RADIUS = 0.0282`; this value is embedded in the precomputed `WalkableArea` rects and must agree across every location where it appears in the codebase (shared `World.ts`, `HeartSprite.tsx`, `CapsuleFallback.tsx`, and the map specs).
- Y-layer render order: −0.01 (background plane) < 0 (floor) < 0.002 (vote circle fill) < 0.003 (vote circle border ring) < 0.004 (vote circle label) < 0.005 (outside textures); collapsing these values causes z-fighting.
- Camera rects are authored in room-local coordinates (origin at room centre, same axes as world space) inside each `RoomSpec`. Transition zone corners are authored in room-A-local coordinates inside each `RoomConnection`. Both are converted to world space once at startup by `buildCameraConstraintShapes`, which is called with the map's local-id `roomPositions` (not the scoped-id map).
- Camera damping uses independent exponential time constants for X and Z axes (`DAMPING_X = DAMPING_Z = 0.1` s). At 60 fps this yields ≈ 15 % movement per frame toward the constrained target.
- Scoped room id format is exactly `{mapInstanceId}_{localRoomId}` — a single underscore; local room ids are assumed to contain no underscore collisions with map-instance ids.
- In the current deployment each world has exactly one map instance, and its `mapInstanceId` equals the scenario id (`demo`, `scenario1`, …, `scenario4`). Demo rooms are therefore `demo_room1`, `demo_room2`, `demo_room3`.
- `WorldSpec.visibility` entries inside each map file are authored with **local** ids on both sides of the mapping; `buildMapInstanceArtifacts` scopes both keys and values at map-instance build time.
- `CURRENT_MAP.roomPositions` is keyed by scoped ids; `WorldSpec.rooms[i].id` is a **local** id. JSX lookups therefore build the scoped key on the fly via `` `${CURRENT_MAP.mapInstanceId}_${room.id}` ``.
- `localPlayerPos.roomId` starts as `''` and is replaced with a scoped id the first time `CURRENT_MAP.getRoomAtPosition` resolves to a room; `gameStore.currentRoomId` also defaults to `''` and is written the same way. Consumers must tolerate the empty string.
- A room is considered overlapping iff its floor rectangle intersects another room's floor rectangle in world space. Overlapping rooms are hidden by default on the client unless the player's `currentRoomId` equals the room, or `playerRoomVisibilityOverride[scopedId]` has been set true by the server. Non-overlapping rooms follow the map's default adjacency (`getAdjacentRoomIds`).
- `validateWorldSpec` still runs on each local `WorldSpec` (including that rooms within a single map instance do not overlap); cross-map-instance overlaps are legal and handled by the visibility rules above.
- The camera constraint works for any rectangular grid graph of room connections within a single map instance; rooms within one map must not overlap (enforced by `validateWorldSpec`), but no linear chain structure is assumed.
- `currentRoomId` in the Zustand store lags one frame behind `localPlayerPos.roomId`; use the store for JSX rendering and `localPlayerPos.roomId` for per-frame logic.
